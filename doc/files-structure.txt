Repository layout (workspace relative)
=====================================

app/
  dashboard/                 → Next.js App Router entry for the authenticated shell.
    _components/             → Sidebar, header, onboarding guards, and other chrome pieces.
    WorkspaceProvider.tsx    → React context that exposes the active workspace id for feature modules.
  (landing)/                 → Marketing pages served from the App Router.
  api/                       → Serverless route handlers (Convex bridges, webhooks, etc.).

frontend/
  features/
    documents/               → Document & CMS management experience.
      page/                  → Client entry for the documents dashboard tab.
      views/                 → Page-level views (e.g. Workspace vs Database tabs).
      shared/                → Hooks, facades, and state shared across document views.
    wa/                      → WhatsApp-inspired workspace communication.
      components/            → UI modules grouped by domain (calls, chat, status, etc.).
        calls/               → Calls list + detail views. Uses shared mockData.ts during development.
        navigation/TopBar/   → Responsive conversation header & action rail.
      shared/                → Zustand store, hooks, and repository interfaces.
    canvas/, search/         → Additional application surfaces (not modified in this pass).
  shared/
    layout/view/             → Generic list rendering primitives (table/card/detail).
      RowActions.tsx         → Centralised button renderer used by all list variants.
    layout/view/ViewSwitcher → Adaptive view switcher that composes the primitives above.
    ui/                      → Shared design system elements (atoms/molecules from shadcn/ui).

components/                  → Coarse design system surface (marketing + app overlays).
convex/                      → Convex backend schema, queries, and auth adapters.
hooks/, lib/, scripts/       → Cross-cutting utilities (hooks, helpers, local scripts).
public/, styles/             → Static assets and Tailwind tokens.

doc/                         → Living documentation. Updated files describe architecture, change-log, and task tracking.

Key conventions
---------------
- Feature modules live under `frontend/features/<domain>`. Each domain exposes:
  * `components/` – leaf UI widgets.
  * `views/` – orchestrated screens that wire data sources + components.
  * `shared/` – stores, hooks, and adapters.
- Shared primitives live under `frontend/shared/`. Prefer composing these instead of duplicating layout logic.
- When introducing list-style UIs, use `RowActions` + `TableView` / `CardView` / `DetailListView` instead of custom action button markup.
- WhatsApp tooling reads from Zustand (`useWhatsAppStore`). The calls view now consumes `mockData.ts`; replace this with live data sources when the backend is ready.
- Documentation and architectural guides are colocated in `doc/`. Keep them ASCII and regenerate when feature domains evolve.
